# Account Online/Offline Observability Design

## Goal

Make every Armada account online attempt diagnosable after the fact. When an account gets stuck in VERIFYING, goes ONLINE then quickly OFFLINE, enters PROXY_FAILED, or is rescued by reconnect logic, operators should be able to answer three questions from durable data:

1. Which online attempt was this?
2. What state transitions happened across Armada and the protocol layer?
3. What evidence explains the final offline or failure reason?

The first implementation should focus on the test environment account-online workflow, but the design should be safe to enable in production later.

## Current Baseline

Armada already has a reliable command and state-event path:

- Armada creates protocol command outbox rows with `commandId`, `batchId`, account identity, credential format, proxy id, and source.
- Armada publishes commands to Kafka.
- The protocol layer consumes account online commands and drives Baileys.
- The protocol layer emits `account.state_changed`.
- Armada consumes `account.state_changed` and updates account login state.

The current logs and events are enough to see broad state movement, for example `VERIFYING -> ONLINE`, `ONLINE -> PROXY_FAILED`, or `VERIFYING -> PROXY_FAILED`. They are not enough to fully diagnose every failure because:

- There is no durable `onlineAttemptId` tying command outbox, protocol worker logs, protocol events, Armada account state, and retry attempts together.
- The protocol worker receives `commandId` and `batchId` in the envelope, but the online execution path does not preserve them into account runtime context.
- `account.state_changed` carries state and some raw code data, but not a structured diagnosis payload.
- VERIFYING timeout tells us that no `connection.update open/close` arrived, but not which lower-level phase failed: proxy connect, TLS, HTTP CONNECT, DNS, or WA handshake.
- Proxy failure auto-reonline can allocate the same proxy again after release, so repeated failures can look like separate attempts without an obvious shared cause.

## Non-Goals

This design does not replace the protocol state machine or Baileys. It does not require tracing every network packet. It also does not make online success synchronous; account online remains an async command flow.

## Design Summary

Add a per-online-attempt observability path:

- Armada generates an `onlineAttemptId` for each online request or reonline request.
- The id is written into command outbox payloads and propagated through Kafka to the protocol worker.
- The protocol worker stores attempt context in `AccountManager` while the account is active.
- The protocol layer emits a new diagnostic event when an attempt fails, goes offline, or triggers rescue logic.
- Armada consumes diagnostic events into a durable timeline table.
- Existing `account.state_changed` continues to be the lifecycle event used for state synchronization.

This keeps lifecycle state changes and diagnosis separate. State changes stay compact and operationally critical. Diagnosis events can carry richer evidence without making the state consumer brittle.

## Identity Model

Use a new string id named `onlineAttemptId`.

Recommended format:

```text
oa_<yyyyMMddHHmmss>_<shortRandom>
```

The id must be generated by Armada before creating the protocol command outbox row. That makes the id available in:

- `protocol_command_outbox.payload`
- Kafka command payload
- protocol worker runtime logs
- protocol diagnostic event payload
- Armada durable diagnosis table

`commandId` remains the delivery id for one command message. `batchId` remains the grouping id for a batch operation. `onlineAttemptId` means one logical attempt to bring one account online with one selected proxy and credential snapshot.

For auto-reonline, Armada should create a new `onlineAttemptId` for each retry. The retry event should also carry `previousOnlineAttemptId` when available, so repeated attempts can be chained.

## Command Payload Changes

Extend the Armada command payload for account online:

```json
{
  "accountId": 9,
  "protocolAccountId": "acc_252625852450",
  "credentialFormat": "BAILEYS_AUTH_INFO",
  "proxyId": 4035,
  "source": "batch_online",
  "onlineAttemptId": "oa_20260702101716_x7k9m2",
  "previousOnlineAttemptId": null
}
```

The protocol command envelope already has `commandId` and `batchId`. The protocol worker should copy these envelope fields into account runtime context together with the payload fields.

Runtime context should include:

- `tenantId`
- `accountId`
- `protocolAccountId`
- `onlineAttemptId`
- `previousOnlineAttemptId`
- `commandId`
- `batchId`
- `proxyId`
- `source`
- `workerId`
- `startedAt`

The current protocol online execution path should stop hardcoding source as `api`; it should preserve the source from the command payload.

## Diagnostic Event Contract

Add a protocol event:

```text
account.offline_diagnosed
```

The event should be emitted when the protocol layer has a meaningful diagnosis point:

- VERIFYING times out without any connection update.
- Baileys emits `connection.close` with a raw code or error message.
- Heartbeat declares a stale connection.
- Quick rescue fails.
- Manual offline closes an account.
- Login becomes unrecoverable, such as logged out, device removed, or reauth required.
- The worker releases a runtime slot because of a failure.

Event data:

```json
{
  "tenantId": 1,
  "accountId": 9,
  "protocolAccountId": "acc_252625852450",
  "onlineAttemptId": "oa_20260702101716_x7k9m2",
  "previousOnlineAttemptId": null,
  "commandId": "cmd_xxx",
  "batchId": "batch_xxx",
  "proxyId": 4035,
  "source": "batch_online",
  "from": "VERIFYING",
  "to": "PROXY_FAILED",
  "diagnosisCode": "VERIFY_TIMEOUT_NO_CONNECTION_UPDATE",
  "diagnosisClass": "PROXY_OR_WA_CONNECTIVITY",
  "rawCode": 408,
  "rawReason": "no connection.update open/close before verify timeout",
  "recoverability": "RETRYABLE",
  "actionTaken": "MARK_PROXY_FAILED_RELEASE_SLOT",
  "occurredAt": "2026-07-02T10:18:00.123Z",
  "workerId": "w3",
  "evidence": {
    "connectionField": "connecting",
    "wsOpen": false,
    "lastConnectionUpdateAt": null,
    "verifyTimeoutMs": 30000,
    "lastHeartbeatAt": null,
    "lastProbeRttMs": null
  }
}
```

Evidence must be bounded and sanitized. It must not contain credential data, QR data, cookies, full proxy password, or large stack traces. Long raw reason strings should be truncated before publishing.

## Diagnosis Codes

Start with a fixed enum that covers known cases and leaves room for unknowns:

| Code | Class | Meaning |
| --- | --- | --- |
| `VERIFY_TIMEOUT_NO_CONNECTION_UPDATE` | `PROXY_OR_WA_CONNECTIVITY` | Account stayed in VERIFYING and Baileys did not emit open or close before timeout. |
| `WA_CONNECTION_TERMINATED_428` | `WA_TRANSIENT` | WA/Baileys closed with raw code 428. Usually retryable, but repeated failures with the same proxy are suspicious. |
| `HEARTBEAT_TIMEOUT` | `STALE_CONNECTION` | Protocol heartbeat stopped receiving timely liveness proof. |
| `QUICK_RESCUE_FAILED` | `STALE_CONNECTION` | Rescue path was attempted but did not recover the account. |
| `STALE_HALF_OPEN_WS` | `STALE_CONNECTION` | WebSocket appears open but does not behave as live. |
| `NEED_REAUTH` | `AUTH` | Credential can no longer restore the session without reauth. |
| `LOGGED_OUT` | `AUTH` | WA reported logout or session invalidation. |
| `DEVICE_REMOVED` | `AUTH` | Linked device was removed. |
| `MANUAL_OFFLINE` | `OPERATOR_ACTION` | User or system intentionally took the account offline. |
| `PROXY_CONNECT_ERROR` | `PROXY_NETWORK` | Proxy connection failed with an explicit connect error. |
| `PROXY_CONNECT_TIMEOUT` | `PROXY_NETWORK` | Proxy connection timed out before WA handshake. |
| `PROXY_AUTH_FAILED` | `PROXY_NETWORK` | Proxy rejected authentication or returned an auth-specific failure. |
| `UNKNOWN_OFFLINE` | `UNKNOWN` | Offline happened without enough known evidence. |

The protocol layer should prefer specific codes when evidence is available. If only a raw close code or message is available, use the closest known code and preserve the raw fields in the event.

## Armada Persistence

Add a durable table for timeline entries:

```sql
CREATE TABLE account_online_attempt_log (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  tenant_id BIGINT NOT NULL,
  account_id BIGINT NOT NULL,
  protocol_account_id VARCHAR(128) NOT NULL,
  online_attempt_id VARCHAR(64) NOT NULL,
  previous_online_attempt_id VARCHAR(64) NULL,
  command_id VARCHAR(64) NULL,
  batch_id VARCHAR(64) NULL,
  proxy_id BIGINT NULL,
  source VARCHAR(64) NULL,
  from_state VARCHAR(32) NULL,
  to_state VARCHAR(32) NULL,
  diagnosis_code VARCHAR(64) NOT NULL,
  diagnosis_class VARCHAR(64) NOT NULL,
  raw_code INT NULL,
  raw_reason VARCHAR(512) NULL,
  recoverability VARCHAR(32) NULL,
  action_taken VARCHAR(64) NULL,
  worker_id VARCHAR(64) NULL,
  evidence_json JSON NULL,
  occurred_at DATETIME(3) NOT NULL,
  created_at DATETIME(3) NOT NULL,
  KEY idx_attempt (tenant_id, online_attempt_id),
  KEY idx_account_time (tenant_id, account_id, occurred_at),
  KEY idx_proxy_time (tenant_id, proxy_id, occurred_at),
  KEY idx_code_time (tenant_id, diagnosis_code, occurred_at)
);
```

This table is append-only. It should not drive the account state machine. It is an audit and diagnosis surface.

Retention:

- Test environment: keep 30 days.
- Production default: keep 7 to 14 days for detailed evidence, then archive or delete.
- Keep rows small enough that account online performance does not depend on this table.

## Armada Consumer Behavior

Extend the protocol event consumer to handle `account.offline_diagnosed`:

1. Parse the event envelope.
2. Validate tenant id, account id, protocol account id, and online attempt id.
3. Persist one row into `account_online_attempt_log`.
4. Log and skip malformed diagnostic events without affecting `account.state_changed` processing.

The state synchronization consumer should remain independent. If diagnostic persistence fails, it should not block account state updates.

## Query Surface

Add a small read surface for operators and scripts:

- Query recent attempts by account.
- Query one attempt timeline by `onlineAttemptId`.
- Query recent failures by proxy.
- Query repeated failures by diagnosis code.

For the current support workflow, the most useful output is a compact timeline:

```text
account 252625852450
attempt oa_20260702101716_x7k9m2
proxy 4035

10:17:16 VERIFYING started by batch_online
10:17:26 ONLINE
10:17:30 PROXY_FAILED WA_CONNECTION_TERMINATED_428 rawCode=428
10:18:00 PROXY_FAILED VERIFY_TIMEOUT_NO_CONNECTION_UPDATE rawCode=408
10:18:11 ONLINE
10:22:15 STALE HEARTBEAT_TIMEOUT
10:22:15 QUICK_RESCUE_FAILED
10:22:45 PROXY_FAILED VERIFY_TIMEOUT_NO_CONNECTION_UPDATE
10:23:01 ONLINE
```

The first slice can be a DbTest-friendly mapper query and an internal admin API later. A CLI or Codex skill can use the same query pattern.

## Performance Controls

The mechanism must not add synchronous waits to account online.

Controls:

- Generate `onlineAttemptId` in Armada in memory; no extra network round trip.
- Publish protocol diagnostic events through the existing async event publisher.
- Treat diagnosis persistence as best-effort for observability, separate from state updates.
- Keep Prometheus labels low-cardinality: diagnosis code, class, worker id, and action. Do not use account id or attempt id as metric labels.
- Store bounded evidence JSON. Do not store large logs or stack traces.
- Add indexes only for support queries; avoid wide composite indexes that slow writes.
- Retain detailed rows for a limited period.

Expected impact:

- Online command creation: one extra string id and a few payload fields.
- Protocol runtime: one small context object and occasional event publish on meaningful transitions.
- Database: one insert per diagnostic point, not one insert per heartbeat.

This should be safe for normal online throughput if diagnostic events are emitted only at state boundaries and failure points.

## Rollout Plan

Implement in slices:

1. Armada migration and mapper for `account_online_attempt_log`.
2. Armada `onlineAttemptId` generation and outbox payload propagation.
3. Protocol command parser and account runtime context propagation.
4. Protocol diagnosis helper that maps known close, timeout, heartbeat, rescue, auth, and manual-offline cases into diagnosis codes.
5. Protocol `account.offline_diagnosed` event emission at meaningful diagnosis points.
6. Armada consumer support for `account.offline_diagnosed`.
7. Operator query for account attempt timeline.
8. Optional lower-level proxy instrumentation if VERIFY timeout remains too coarse.

The optional proxy instrumentation should come after the durable attempt timeline exists. It can add phases such as proxy TCP connect, proxy auth, HTTP CONNECT, TLS, and WA handshake when the current libraries expose enough signal.

## Testing Strategy

Armada tests:

- Migration and mapper DbTest for inserting and querying diagnosis rows.
- Outbox payload unit tests proving `onlineAttemptId` and `previousOnlineAttemptId` are present.
- Event consumer tests for valid, malformed, duplicate, and unknown-code diagnostic events.
- Query tests for recent account timeline and proxy failure lookup.

Protocol tests:

- Command parser test proving envelope `commandId` and `batchId` plus payload attempt fields enter runtime context.
- `AccountManager` tests for VERIFY timeout diagnosis.
- `AccountManager` tests for 428 connection termination diagnosis.
- Heartbeat stale and quick-rescue-failed diagnosis tests.
- Manual offline and auth-related diagnosis tests where current code has deterministic hooks.
- Event publisher test proving evidence is sanitized and bounded.

Integration checks:

- Bring a test account online and confirm one `onlineAttemptId` is visible in Armada outbox, protocol logs, protocol event payload, and Armada diagnosis table.
- Force VERIFY timeout and confirm diagnosis code `VERIFY_TIMEOUT_NO_CONNECTION_UPDATE`.
- Confirm `account.state_changed` processing still updates login state even if diagnostic persistence fails.

## Operational Use

For account `252625852450`, the future workflow should be:

1. Find the latest Armada account row and protocol account id.
2. Query recent `account_online_attempt_log` rows by account id.
3. Pick the latest `onlineAttemptId`.
4. Read the timeline and diagnosis codes.
5. If repeated failures use the same proxy id, inspect proxy allocation and release behavior.
6. If diagnosis is VERIFY timeout, check whether lower-level proxy instrumentation is available; otherwise classify as proxy-or-WA connectivity with insufficient lower-level evidence.

This gives support a durable answer even when PM2 logs have rotated or the worker has restarted.

## Decisions

These choices are fixed for the first implementation:

- Use a new `account.offline_diagnosed` event instead of overloading `account.state_changed`.
- Keep diagnostic persistence separate from account state synchronization.
- Generate `onlineAttemptId` in Armada, not the protocol layer.
- Do not put account id or attempt id into Prometheus labels.
- Keep low-level proxy phase instrumentation as a later slice.
