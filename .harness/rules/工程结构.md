# 工程结构规范

armada 仓库的目录与包边界规范。新增代码必须落到正确的**业务域**与**层**,违反依赖方向的改动会被 ArchUnit 挡下。

> 本规范描述 armada(单工程、按业务域分包)。完整设计与决策见 `armada/docs/design.md`。

## 顶层结构
| 路径 | 内容 |
|------|------|
| `armada-api/` | 后端,Java 17 + Spring Boot 3.3.5,**单 Maven 工程**,包根 `com.armada` |
| `armada-admin-web/` | 云控前端(L1 最高控 + L2 二级代理,按 role 控菜单) |
| `armada-saas-web/` | SaaS 前端(L3 三级代理,承载具体业务) |
| `armada-deploy/` | 测试服一键部署脚本 |

## 后端:单工程 · 按业务域分包(`com.armada`)
不再是多模块。一个工程内按**业务域**分包,域内 controller-service-mapper。顶层 = 9 个域(6 业务 + 3 基础设施):

```
com/armada/
  account/    账号:导入/列表/6态生命周期/凭据/分组/协议号导出
  group/      群组/群链接/导入链接
  task/       拉群+进群任务、调度引擎、补拉手、水军
  marketing/  群组营销任务/模板/发送引擎
  resource/   代理IP、料子/物料
  admin/      后台:租户/用户/角色/权限/BI/操作日志
  platform/   基础设施:协议层/Kafka/文件上传/持久化
  shared/     纯基础:异常/分页/响应/租户上下文/工具/枚举
  boot/       启动 + 全局装配
```
- 顶层 = 业务域,几乎不增(只有整块全新业务才加);菜单都长在域**内部**。

## 业务域内部结构
6 个业务域(account/group/task/marketing/resource/admin)统一:
```
<业务域>/
  controller/    XxxController(按菜单命名,如 AccountListController)
  service/       XxxService 接口
    impl/        XxxServiceImpl 实现
  mapper/        XxxMapper(MyBatis 接口,标 @Mapper,按表命名)
  converter/     XxxConverter(MapStruct @Mapper 转换接口)
  model/
    entity/      实体(普通类 + getter/setter,无 Lombok,裸名如 Account)
    dto/         入参:XxxQuery(可变 class extends PageQuery,读) / XxxDTO(record,写)
    vo/          出参:XxxVO(record)
    enums/       本功能专属枚举/状态码(如 JoinTaskStatus/DistributionMode);非表实体,不放 entity/
```
- **枚举归属**:功能专属枚举/状态码放**域内** `model/enums/`;跨域公共枚举放 `shared/enums`。二者皆非实体,**禁放 `model/entity/`**。
- **无 Repository 层**:Service 直接调 Mapper(Mapper 接口即数据访问缝)。
- controller / service 按**菜单**命名;mapper / entity 按**表**命名(一菜单可读写多表,一表可被多菜单用)。
- 域内菜单特别多(>8–10)时,**只在该域**引入功能子包,不全局预先嵌。

## 基础设施域内部结构
```
shared/    tenant(TenantContext / @IgnoreTenant) · exception(BusinessException/ErrorCode) · response(ApiResponse/PageResult) · paging(PageQuery) · enums · util
platform/  protocol(协议层 laqunxitong client + 防腐层) · kafka(消费/生产,含状态回写 handler) · proxy(运行时 IP 出口解析适配器) · upload · persistence(MyBatis 配置 + 租户拦截器 TenantLineConfig/MybatisPlusConfig + IGNORED_TABLES)
boot/      Application · config(所有 @Configuration) · web(全局异常处理器 + 鉴权拦截器,解析并写入 TenantContext)
```

### `platform/protocol` 目录规划
协议层接口按能力族拆分,禁止做一个大而全的 `ProtocolClient`,也禁止把 HTTP DTO 泄漏进业务域。

```
platform/protocol/
  port/       account/group/message/contact/profile/admin 等能力接口
  model/      armada 内部协议语义模型
  http/       HTTP 适配实现 + dto + ProtocolHttpExecutor
  config/     协议层配置
  exception/  协议异常与错误码
```

- 业务域只能依赖 `port` / `model`,禁止依赖 `http` / `http.dto`;
- `port` 按业务能力分组,如 account 下再拆 lifecycle/status/auth/proxy;
- `http` 负责 URL、header、超时、错误映射、日志脱敏与协议契约兼容;
- 首期上线只落 account lifecycle,后续按能力族增量补。

## Mapper XML / Flyway / 扫描
- Mapper XML:`armada-api/src/main/resources/mapper/<业务域>/XxxMapper.xml`,namespace 指接口全限定名;
- Mapper 扫描:每个 mapper 接口标 `@Mapper`(`org.apache.ibatis.annotations.Mapper`)+ 单条 `@MapperScan(basePackages="com.armada", annotationClass=Mapper.class)`(只认 MyBatis 的 @Mapper,不误抓 service / MapStruct converter);
- Flyway 迁移:`armada-api/src/main/resources/db/migration/`(`V<nnn>__<snake_desc>.sql`)。

## 依赖方向(ArchUnit 强制)
```
shared ← platform ← {account, group, task, marketing, resource, admin} ← boot
```
- `shared`:不依赖 `platform` 与任何业务域;
- `platform`:可依赖 `shared`,不依赖业务域;
- 业务域:可依赖 `shared` / `platform`;**跨业务域只能调对方 Service**,禁碰对方 controller / mapper / entity;
- `boot`:装配层,可依赖全部。

## 不可违反的边界
- 跨域调用必须走对方 **Service**,禁直接依赖其他域的 controller / mapper / entity;
- Controller 禁直接访问 Mapper,必须经 Service;
- 跨租户豁免 `@IgnoreTenant` 只能出现在 `admin` 域,且必须声明非空 reason;
- 文件 / 方法保持足够小,让人 / Agent 一次读懂(类 ≤800、方法 ≤100,见编码规范)。

## 命名规范
| Type | Convention | Example |
|------|------------|---------|
| Java Class | PascalCase | `AccountListService` |
| Java Method / Field | camelCase | `getById` / `tenantId` |
| Java Package | all lowercase | `com.armada.account` |
| REST Endpoint | kebab-case | `/api/v1/pull-tasks` |
| DB Table / Column | snake_case | `task_row` / `tenant_id` |
| Vue Component | PascalCase | `AccountList.vue` |
| Vue Composable | use + camelCase | `usePagedList.ts` |
| Pinia Store | use + camelCase + Store | `useAccountStore.ts` |
| 实体 entity | 裸名(无后缀) | `Account` |
| Mapper 接口 | `<Table>Mapper` | `AccountMapper` |
| Mapper XML | 与接口同名,按域放 | `mapper/account/AccountMapper.xml` |
| MapStruct 转换器 | `<X>Converter` | `AccountConverter` |
| 入参(写,@RequestBody) | `<X>DTO`(record) | `AccountCreateDTO` |
| 入参(读列表,@ModelAttribute) | `<X>Query`(class extends PageQuery) | `AccountListQuery` |
| 出参 | `<X>VO`(record) | `AccountListVO` |
| Flyway 迁移 | `V<nnn>__<snake_desc>.sql` | `V001__account_init.sql` |

## 红线规则
1. 🔴 顶层包名固定 `com.armada`,禁止其他根包名。
2. 🔴 依赖方向不可反转:`shared ← platform ← 业务域 ← boot`;业务域之间禁止串门(只能走对方 `Service`)。
3. 🔴 分层不可绕过:`Controller → Service → Mapper`,Controller 禁直连 Mapper。
4. 🔴 跨业务域调用必须走对方 `Service`,禁直接依赖其 controller / mapper / entity。
5. 🔴 一个菜单 / 功能的类待在其所属业务域,禁止散落或跨域放置。
6. 🔴 Mapper XML 放 `resources/mapper/<域>`、Flyway 放 `resources/db/migration`,禁止散落。
7. 🔴 **无 Repository 层**:Service 直接调 Mapper;禁止重新引入转发型 Repository。
8. 🔴 传输对象:entity=普通类(`model/entity`,无 Lombok);`Query`=可变 class extends PageQuery;`DTO`/`VO`=record;转换用 MapStruct(`converter/`)。
9. 🔴 前端页面按所属端放置:L1/L2 → `armada-admin-web`,L3 → `armada-saas-web`。
