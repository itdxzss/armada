# 编码规范

wheel 的编码标准与约定。本文件是正式规范，违反任一条都视为缺陷。

## 反屎山行为约束（改代码前必读，最高优先级）

> 屎山几乎全部来自这几个动作性坏习惯，而非"不会写好代码"。下面每条都是**可自检的硬规则**，
> 改任何代码前对照执行；与本节冲突的"图省事"做法一律不允许。

1. **先读后写（复用优先）**：动手前先用 codegraph / grep 找同类已有实现，
   能复用就复用、能扩展就扩展、否则照它的写法来。新写一个方法/工具/类之前，
   必须能回答"现有的 X 为什么不能用"——答不上来就用现有的，不要并行造第二套。
2. **改行为 = 删旧路径（只增不删是头号病）**：修改逻辑时删干净老路径，
   不留死代码、不留注释掉的代码、不留没人调的参数/分支，**不加"向后兼容"构造器或重载来绕开改调用方**。
   - ❌ 新增第 N 个重载 / 加一个 `boolean useNew` 开关来回避改那十几处调用
   - ✅ 直接改调用方（有测试兜底，放心改）；确实需要兼容时必须在 PR 里写明理由
3. **范围克制（diff 最小）**：只做被要求的事。不顺手重构无关代码、不加没要求的功能/参数/配置项/抽象。
   一次改动的 diff 只应包含本任务相关内容。
4. **不臆造抽象**：没有 ≥3 个真实调用点，不抽公共方法 / 基类 / 泛型 / 配置项。
   不为"将来可能用到"的场景提前设计（呼应「代码结构与设计约束」第 11 条）。
5. **匹配周边代码**：编辑某文件前先读该文件 + 1~2 个相邻同类文件，
   命名、分层、错误处理、日志风格一律照它们来，禁止在局部引入新风格/新模式。
6. **并发会话不整文件重写**：本仓存在多 agent 同 worktree 并行（同一批文件随时被改）。
   改文件前先 re-read 最新内容，**只做最小 diff，禁止整文件重写**；提交前 `git diff` 确认没覆盖他人在途改动。
7. **可恢复错误必须显式抛业务异常**：面向运营可恢复的错误抛 `BusinessException`（业务码），
   不要落到 HTTP 200 被前端当成功吞掉（呼应「Spring 编码准则」第 5 条与「红线」第 4 条）。

### 提交前反屎山自检清单（逐条打勾，做不到不许称"完成"）

- [ ] 没有新增重复逻辑（动手前搜过同类实现）
- [ ] 改了行为的地方，旧路径 / 死代码 / 临时兼容 shim 已删干净
- [ ] 没加没被要求的参数 / 抽象 / 功能 / 配置项
- [ ] 新代码风格与周边一致，未引入新模式
- [ ] diff 只含本任务相关改动，无顺手重构、无无关文件
- [ ] 测试先红后绿，且按本文件「测试代码」节真库验证（非 mock）

## Java 基础风格

1. 使用 JDK 17 语法，禁止引入更高版本语法或依赖。
2. 缩进使用 4 个空格；禁止 tab。
3. 类名使用 `UpperCamelCase`，方法/字段/局部变量使用 `lowerCamelCase`，常量使用 `UPPER_SNAKE_CASE`。
4. import 不使用通配符；顺序保持：项目内/第三方/JDK/Spring，提交前按 IDE 自动整理。
5. 方法保持短小，超过约 80 行应优先拆分；不要为了“抽象”拆出无意义 helper。
6. 禁止在生产代码里留下 `TODO 接库`、硬编码假数据、临时 mock 返回。

## Java 注释规范

1. 类、字段、公开方法、接口方法必须使用 Javadoc 注释，说明业务含义、参数、返回值和异常。
2. Controller / Service / Repository 的公开方法注释必须说明业务目的，禁止只复述方法名。
3. 方法内部注释只解释复杂业务规则、状态流转、租户隔离、SQL 特殊处理、协议层约束等“为什么”，禁止无意义注释。
4. 枚举值、状态码、错误码、任务类型、账号类型必须逐项注释业务含义。
5. 修改代码时必须同步修改注释，注释与实现不一致视为缺陷。
6. 禁止提交大段注释掉的废代码；确需保留时必须写明原因和清理条件。

### 注释示例

Service 方法注释要说明业务边界、事务原因和异常语义：

```java
/**
 * 创建租户进群任务。
 *
 * <p>该方法只负责把任务定义、群链接、执行账号等信息落库，并生成待执行明细行。
 * 真正调用协议层执行进群由后台 worker 在事务提交后触发，避免 worker 读不到刚插入的任务数据。</p>
 *
 * @param tenantId 当前租户 ID，必须来自租户上下文，禁止前端传入覆盖
 * @param request 创建任务请求，包含任务名称、群链接、执行账号和分配策略
 * @return 创建后的任务详情
 * @throws BusinessException 当群链接为空、账号不存在、任务参数非法时抛出
 */
@Transactional(rollbackFor = Exception.class)
public JoinTaskDetail createJoinTask(Long tenantId, CreateJoinTaskDTO request) {
    validateCreateRequest(request);

    // 协议层执行依赖任务明细行，必须先完成 DB 落库，再由 afterCommit 触发 worker。
    JoinTask task = saveTaskDraft(tenantId, request);
    List<JoinTaskRow> rows = buildPendingRows(tenantId, task.getId(), request);
    joinTaskMapper.batchInsertRows(rows);

    afterCommitExecutor.execute(() -> joinTaskWorker.startAsync(tenantId, task.getId()));

    return toDetail(task, rows);
}
```

枚举注释要逐项解释业务含义：

```java
/**
 * WhatsApp 账号业务类型。
 */
public enum AccountType {

    /**
     * 平台自有账号池账号，可分配给租户使用。
     */
    PLATFORM_POOL,

    /**
     * 租户自行导入账号，归属当前 tenant_id。
     */
    TENANT_OWN,

    /**
     * 协议层临时回传但尚未完成归档的账号。
     */
    PROTOCOL_PENDING
}
```

不要写只复述代码的废话注释：

```java
// ❌ 设置用户名
user.setName(name);

// ✅ 协议层只接受 E.164 格式手机号，入库前统一归一化。
String normalizedPhone = phoneNormalizer.normalize(phone);
```

## Spring 编码准则

1. Service 优先构造器注入；字段注入禁止用于新代码。
2. Controller 只做参数接收、权限/上下文衔接、响应组装；业务规则放 Service。
3. Mapper 只做数据访问（SQL），不承载业务流程；**无 Repository 层**，Service 直接调 Mapper。
4. 事务边界放在 Service 层；需要事务的方法显式标注 `@Transactional`。
5. 异常统一使用项目已有 `BusinessException` / `ErrorCodes`，不要直接向前端抛裸 RuntimeException。
6. 日志使用 slf4j，占位符 `{}`；禁止拼接敏感信息，账号、token、creds、代理密码必须脱敏。

## 代码结构与设计约束

1. java代码单个方法除注释外不超过100行。
2. java单个类除注释外不超过800行。
3. java单个方法参数不超过5个,超过必须封装为对象。
4. 单个方法的圈复杂度不超过10。
5. 嵌套层级不超过  3 层

## 传输对象（entity / Query / DTO / VO）

armada 口径,均放 `<业务域>/model/`（详见 design.md §5、工程结构.md）。

1. **entity**（库表实体）放 `model/entity/`，裸名（如 `Account`），**普通类 + getter/setter（无 Lombok）**，Mapper 直出实体；一域内多功能共用的实体放该域 `model/entity/`。
2. **VO**（出参）放 `model/vo/`，`record`，命名 `<X>VO`，**字段 camelCase；JSON wire 即 camelCase**（全局无 Jackson 命名策略,默认 camelCase）。统一 `VO`，废弃 `Vo`/`View`/`Response` 三套并行。
3. **DTO**（写入参，@RequestBody）放 `model/dto/`，`record`，命名 `<X>DTO`，**字段 camelCase；JSON wire 即 camelCase**。`@RequestParam`/`@ModelAttribute` 入参名也用 camelCase(与 JSON 契约一致)。
4. **Query**（读/列表入参，@ModelAttribute）放 `model/dto/`，命名 `<X>Query`，**必须可变 class 且 `extends PageQuery`**（@ModelAttribute 走 setter，不认 record）；参数封装规则见「代码结构与设计约束」第 3 条。
5. **转换用 MapStruct**：`<X>Converter`（`@Mapper(componentModel="spring")`）放 `<业务域>/converter/`，注入 Service；禁手写大段 set、禁 `BeanUtils` 反射拷贝。
6. **无 Repository 层**：Service 直接调 Mapper。

## MyBatis / 数据访问

1. 生产代码只通过 Mapper 查真实 MySQL（无 Repository 层）。
2. 分页、筛选、count 必须 SQL 下推，禁止 load-all 后 Java 内存分页。
3. Mapper XML 修改后必须跑 XML 校验或真库 DbTest。
4. `FOR UPDATE` + `LIMIT` 的 mapper 方法必须处理租户拦截器问题，按 AGENTS.md 红线执行。
5. 分页接口对外统一返回 `com.armada.shared.response.PageResult<T>`，**禁止自造分页返回类**：
   - 用工厂 `PageResult.of(list, page, pageSize, total)` 构造，`totalPages` 自动推导，不要手算。
   - 入参用 `page`（默认 1）/ `pageSize`（默认 10）；序列化字段为 camelCase（`pageSize`/`totalPages`）。
   - `PageResponse` 仅限 service 层内部周转，出 Controller 前必须转成 `PageResult`。

## 测试代码

1. Mockito/mock 仅允许在 `src/test`。
2. Service 单测可以 mock 依赖；Mapper/SQL/Flyway 必须真库验证。
3. 回归事故必须补测试：account_type 冻结、租户隔离、XML 解析、分页下推等。

## 禁止事项

1. 禁止新增 `InMemory*` / `Fake*` / `Stub*` 业务实现进入 `src/main/java`。
2. 禁止 Controller/Service 返回硬编码 demo 数据。
3. 禁止擅自升级 JDK、Spring Boot、Maven 主版本。




## 数据访问（铁律，详见 CLAUDE.md）
- 生产代码只查真库 MyBatis，**禁 mock / 假数据 / 内存兜底 / 内存分页**。
- 分页 + count + 筛选必须 SQL 下推；新数据顺序：Flyway 迁移 → Mapper 接口 + XML → Service。
- `FOR UPDATE` + `LIMIT` 的 mapper 方法必须 `@InterceptorIgnore(tenantLine="true")`
  （租户拦截器 JSqlParser 重写会产生 MySQL 语法错；只有真库 DbTest 抓得到）。
- 改 mapper XML 必须先 `xmllint` / 真库 DbTest 再部署（裸 `<>` 运行时炸 → crash-loop 全站 502）。

## 业务口径（回归测试已锁，别改回）
- `account_type` 导入即冻结，任何操作不得改写。

## 前端
- Vue 3 必须 `<script setup>`，禁 Options API。
- 新弹框遮罩 z-index 必须 < 30000（自绘下拉层级，否则弹框内 select 点击无反应）。
- API 响应统一 `{code, message, data}` 结构。

## 通用
- 中文注释清晰、日志完整便于排查；禁 `System.out.println` / `printStackTrace`。
- deploy 脚本 `$VAR` 后紧跟全角符号一律写 `${VAR}`（非 UTF8 变量名崩溃）。

## 红线规则
1. 🔴 禁止魔法值/硬编码 ，如直接出现数字、字符串常量，如 if (status == 1) 使用枚举或常量类（public static final）， 使用枚举或常量类（public static final）来替代。
2. 🔴 禁止重复代码（DRY），  如三段以上相同/相似的逻辑 ， 使用提取为公共方法或工具类。
3. 🔴 禁止空 catch 块，  如catch (Exception e) {} 不做任何处理 ， 使用必须记录日志或重新抛出业务异常。
4. 🔴 禁止返回 null ， 如方法返回 null 导致调用方 NPE ， 使用 Optional 或返回空集合/空对象。
5. 🔴 禁止System.out.println()， 如在生产代码中使用  使用 @Slf4j + log.info()
6. 🔴 禁止 AI 一次性生成大块不可维护代码：单次改动必须聚焦一个明确目标，禁止顺手重构无关文件、批量搬运逻辑、堆砌重复代码。
7. 🔴 禁止“能跑就行”的胶水代码：新增逻辑必须符合现有分层、命名、异常、日志、事务、数据访问规范，不能绕过 `Service` / `Repository` / `Mapper` 边界。
8. 🔴 禁止把复杂业务写进一个超大方法/超大类：超过方法/类长度、参数数量、嵌套层级、圈复杂度限制时，必须先拆分职责，再实现。
9. 🔴 禁止无上下文改代码：改动前必须先阅读相关 Controller / Service / Repository / Mapper / DTO / 测试，沿用项目已有模式，禁止凭空创造新风格。
10. 🔴 禁止无验证交付：涉及业务逻辑、SQL、权限、租户隔离、状态流转的改动，必须补充或更新测试；无法测试时必须说明风险和人工验证步骤。
11. 🔴 禁止注释与实现不一致：改业务规则、参数、返回值、异常、状态流转、SQL 特殊处理时，必须同步更新 Javadoc 和关键行内注释。
