## 1. 通用契约(所有页面公共,先读这一节)

### 1.1 响应信封(关键)
- 结构:`{ code, message, data }`。
- **`code=0` 表示成功**(不是 200);业务错误返非 0(意外错误 `50000`)。
- **HTTP 状态恒为 200**:业务错误也走 200(`GlobalExceptionHandler` 把 `BusinessException` 转成 `code≠0` 的 body),只有传输层真崩才非 200。
- 前端判成功:**看 `body.code === 0`**,不看 HTTP 200,不看布尔 `success`(后端无此字段)。
```
成功:  HTTP 200  { "code": 0,     "message": "ok",     "data": {...} }
失败:  HTTP 200  { "code": 50000, "message": "系统繁忙", "data": null }
```
> 口径与 wheel 后端一致(wheel `ApiResponse.ok` 也用 `0=成功`),无需改后端。

### 1.2 分页
- 请求:`page`(默认 1)、`pageSize`(默认 10,上限 500),camelCase。
- 响应:`{ list, page, pageSize, total, totalPages }` —— **数组键是 `list`**(不是 `records`/`content`)。`PageResult<T>` 包在 `ApiResponse.data` 里。

### 1.3 字段命名 / 时间
- 出入参全 camelCase(0624 翻转,删了全局 Jackson SNAKE_CASE)。
- 时间字段值是 **epoch 毫秒(Long,UTC 时刻)**,字段名仍 camelCase(如 `createdAt`);前端用 `new Date(ms)` 转 Asia/Shanghai 展示。

### 1.4 租户解析(先测方案,JWT 后置)
- **后端**:加一个拦截器,读请求头 `X-Tenant-Code` → 查租户表得 `tenantId` → `TenantContext.set(tenantId)`;`afterCompletion` 里 `clear()`(防 ThreadLocal 泄漏)。**不验 JWT**。
- **极简登录**:账号密码校验通过 → 返回该用户的租户码(不签 JWT)。
- **前端**:登录后存租户码,之后每个请求自动带 `X-Tenant-Code` 头。
- **升级 JWT 时**:把"租户码从 header 来"换成"从 token 解",拦截器骨架不动;前端把存租户码换成存 token。
> 参照 wheel `TenantContextInterceptor`(它的 tenantId 来自 JWT,`X-Tenant-Code` 仅做一致性校验;armada 先测版把来源直接换成 header)。

### 1.5 部署 / 跨域(nginx 同源,**不靠 CORS、不靠 vite 代理**)
- 照搬 wheel 测试环境:docker-compose 三容器 `mysql + backend(仅内网 expose 8080) + nginx`。
- nginx 同一个 server 块里:① 把前端打包出的 `dist` 当静态文件托管在根路径;② `location /api/ { proxy_pass http://backend:8080; }` 反代到后端。→ 页面与 `/api` **同源**,不触发跨域预检。
- **后端不加任何 CORS 配置**;前端 axios `baseURL` 设 `/`(相对路径);`vite.config` 的 `proxy` 仅本地 `npm run dev` 用,打包产物不带它。

### 1.6 前端对接层改造(公共)
- `utils/http` 响应拦截器:按 `code===0` **集中拆包**(成功返 `data`,非 0 用 `message` 报错),并保留 401/未授权处理。
- api 类型从 `{ success:boolean, data, message? }` 改为 `{ code:number, message?:string, data:T }`。
- 请求拦截器补 `X-Tenant-Code` 注入;`baseURL='/'`。

## 3. 导入链接(分组 → 批次 → 链接)

### 3.1 数据模型(V003 四表,均软删 + 租户隔离)
- `group_link_label`(**WS 链接分组**):name/region/remark…
- `group_link_import_batch`(**批次**):label_id、batch_name、source_file_name、total/inserted/adopted/skipped/failed_rows、created_at。
- `group_link`(**链接**):link_url、group_name、label_id(所属分组)、import_batch_id(来源批次)…
- `group_link_import_detail`(**明细**):batch_id、line_no、raw_url、result(1 成功 2 收编 3 批内重复 4 格式错误)、fail_reason…

层级:`label →(label_id)→ batch`、`label →(label_id)→ link`、`link →(import_batch_id)→ batch`、`detail →(batch_id)→ batch`。即 **一分组 → 多批次 → 多链接**,模型已建好。

### 3.2 页面三层
1. **主表 = 分组列表**(`GET /api/group-link-labels`):分组名 / 国家 / 链接数(`linkCount` 已有) / **成功 / 失败** / 创建时间。
   - 成功/失败现在就能算(不改表):按 `label_id` 汇总批次计数器 —— `成功=SUM(inserted+adopted)`、`失败=SUM(skipped+failed)`。语义="导入进该分组的链接成功/失败多少"。
2. **下钻 = 批次列表**(该分组的批次):批次名 / 来源文件 / 总数 / 成功 / 失败 / 导入时间。
   - ⚠️ 前端当前那张"批次列"表(来源文件/总/成功/失败/导入时间)其实就是**这一层**的列,挪下来即可。
3. **再下钻 = 链接列表**(`GET /api/group-links?labelId=`,可加 `batchId` 过滤)。

### 3.3 编辑换分组 = 已有的 `POST /api/group-links/migrate`
只改链接的 `label_id`,**不动批次、链接保留原来源文件名**——正合"链接不换文件、但能换 WS 分组"。✅ 不用新做。

### 3.4 后端要做
- 分组列表 VO 加 2 个字段(成功/失败)+ `GroupLinkLabelMapper.selectPage` 加 2 个子查询。
- **批次列表读接口(唯一全新一块)**:`GroupLinkImportBatchMapper.selectPageByLabel/countByLabel`(表字段已齐)+ service + `GET /api/group-link-imports/batches?labelId=` + 批次 VO。
- (可选)`/api/group-links` 加 `batchId` 过滤,看单批次的链接。

## 4. 营销模板

| 方法 | 路径 | 说明 |
|---|---|---|
| GET | `/api/marketing-templates` | 列表 |
| POST | `/api/marketing-templates` | 新建 |
| PUT | `/api/marketing-templates/{id}` | 更新 |
| POST | `/api/marketing-templates/{id}/clone` | 复制 |

**字段对照与决策**(`MarketingTemplateVO`:id/templateName/linkMode/textType/imageFileId/content/bodyText/buttons/promotionLink/remark/createdAt/updatedAt):

| 项 | 决策 | 谁改 |
|---|---|---|
| 超链模式 | 前端对齐后端(用 `linkMode` 数字 1 普通 / 2 按钮) | 前端 |
| 列"文本类型" | 对齐后端真 `textType` 字段(现错绑了 linkMode) | 前端 |
| 标题 / 正文 | 前端对齐后端(`content`=标题,正文 `text`→`bodyText`) | 前端 |
| 引用任务数 | **后端加 `referenceTaskCount` 字段**(真值依赖营销任务模块,先恒 0,见 TODO #4) | 后端 |
| 图片 | 走**本地盘 + 按 fileId 流式 GET**(见下),别抄 wheel 的 OSS 预签名 | 后端 |
| 按钮枚举 | **后端对齐前端** `{ type:'link'|'phone'|'copy'|'quick', label, value }` | 后端 |
| 启用/停止 | 先不做(见 TODO #1) | — |

**图片**:armada 当前 `image_file_id` 是悬空 bigint,无文件表 / 无上传下载端点 / 无存储配置。最小实现:
- Flyway V004 建 `marketing_image` 表(id/tenant_id/filename/content_type/size/storage_path/created_at)。
- `application.yml` 加 `armada.upload.dir` + multipart 限制(≤500KB)。
- `POST /api/marketing-images`(MultipartFile)→ 写本地盘 → 返回 `fileId`;`GET /api/marketing-images/{fileId}` → 按 `storage_path` 流式吐字节(`Content-Type`)。
- 前端 `<img src="/api/marketing-images/{imageFileId}">`;VO 可加派生 `imageUrl`。

**按钮枚举改法**:后端现为 `MessageButton{ ButtonType type, text, param }`,枚举 `LINK_JUMP/COPY_CONTENT/QUICK_REPLY`。改:`text→label`、`param→value`,枚举值 → `link/copy/quick`,**并新增第 4 个 `phone`(点击拨号,后端现无对应)**。buttons 在表里存不透明 JSON 文本,改名零迁移;测试数据可直接清。当前无发送方读它,**零运行时风险**(见 TODO #3)。
